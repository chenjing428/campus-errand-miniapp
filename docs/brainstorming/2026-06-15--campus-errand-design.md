# 校园跑腿小程序设计文档

> 文档日期：2026-06-15  
> 文档状态：MVP 设计（标准版）  
> 技术方案：uni-app + uniCloud + 微信支付

---

## 1. 项目背景与目标

### 1.1 背景

在校园场景中，学生经常遇到以下需求：
- 快递到达驿站但上课/外出无法及时取件
- 外卖送到校门口但宿舍区不让进
- 需要把文件/物品送到另一栋宿舍楼或教学楼

### 1.2 项目目标

构建一个校园内部的 C2C 跑腿服务平台：
- 发单者：快速发布跑腿需求，实时追踪订单状态
- 接单者：利用碎片时间接单赚钱
- 平台：提供撮合、支付担保、信用评价、后台管理

### 1.3 成功标准

- 发单到接单平均响应时间 < 30 分钟
- 订单完成率 > 85%
- 纠纷率 < 5%
- 支持单校范围内稳定运营

## 2. 技术架构

### 2.1 选型

| 层级 | 技术 | 说明 |
|---|---|---|
| 小程序前端 | uni-app | Vue 语法，可扩展 H5/App |
| 云函数 | uniCloud | Serverless，免运维 |
| 数据库 | uniCloud 云数据库 | 类 MongoDB，按集合存储 |
| 文件存储 | uniCloud 云存储 | 头像、证件照等 |
| 支付 | 微信支付 + uni-pay | 平台担保模式 |
| 后台管理 | uni-admin | 基于 uni-app 的后台框架 |

### 2.3 取舍分析

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| uni-app + uniCloud | 开发快、免运维、成本低 | 复杂业务需拆分云函数 | ✅ 采用 |
| uni-app + 自建后端 | 控制力最强 | 运维成本高 | 长期可考虑 |
| 微信小程序原生 | 官方支持好 | 扩展性差 | 不推荐 |

## 3. 用户角色

### 3.1 角色定义

| 角色 | 说明 | 权限 |
|---|---|---|
| 普通用户 | 注册并绑定学号的学生 | 发单、查看订单、接单（仅限普通单） |
| 认证骑手 | 通过实名/学生证认证的接单者 | 可接所有类型订单，包括高价值单 |
| 管理员 | 平台运营人员 | 用户管理、订单监控、提现审核、违规处理 |

### 3.2 认证流程

```
微信登录
  │
  ▼
获取 openid
  │
  ▼
绑定手机号
  │
  ▼
绑定学号 + 姓名 + 学校
  │
  ▼
普通用户（可发单、可接普通单）
  │
  ▼（申请认证骑手）
上传学生证/身份证
  │
  ▼
等待管理员审核
  │
  ▼
认证骑手
```

## 4. 服务品类与计价

### 4.1 服务品类

| 品类 | 适用场景 | 特殊字段 |
|---|---|---|
| 快递代取 | 取快递并送到宿舍 | 快递点、取件码、快递大小 |
| 外卖代取 | 校门口取外卖并送到宿舍 | 外卖平台、取餐点、送达时间 |
| 文件代送 | 校内文件/小物品递送 | 取件地址、收件地址、物品描述 |

### 4.2 计价规则

**基础费用 + 附加费用**

| 费用项 | 说明 | 默认值 |
|---|---|---|
| 基础跑腿费 | 每单起步价 | 2 元 |
| 重量费 | 超过 1kg 后每公斤加收 | 1 元/kg |
| 楼层费 | 无电梯且楼层较高时加收 | 0.5 元/层 |
| 距离费 | 取送点距离较远时加收 | 0.5 元/100m |
| 加急费 | 要求快速送达 | 2 元 |
| 小费 | 发单者自愿追加 | 任意 |
| 平台服务费 | 平台抽成比例 | 5% - 10% |

> 注意：具体价格可在后台配置，MVP 阶段支持固定规则即可。

## 5. 核心流程

### 5.1 发单流程

```
选择服务类型
  │
  ▼
填写取货/送货信息
  │
  ▼
系统自动计算费用
  │
  ▼
发单者确认并支付
  │
  ▼
订单进入"待接单"状态
  │
  ▼
推送通知给附近/在线骑手
```

### 5.2 接单流程

```
接单者浏览订单列表
  │
  ▼
抢单/接单
  │
  ▼
订单状态变为"已接单"
  │
  ▼
接单者前往取货点
  │
  ▼
取货后点击"已取货"
  │
  ▼
送达后点击"已送达"
  │
  ▼
发单者确认收货
  │
  ▼
订单完成，款项结算给骑手
```

### 5.3 订单状态机

```
待支付 ──支付成功──► 待接单 ──接单──► 已接单 ──取货──► 配送中 ──送达──► 待确认 ──确认──► 已完成
   │                    │            │            │            │           │
   │                    │            │            │            │           │
   ▼                    ▼            ▼            ▼            ▼           ▼
已取消               已取消       已取消        已取消        已取消      已取消/申诉
```

状态说明：

| 状态 | 说明 |
|---|---|
| 待支付 | 已下单未付款 |
| 待接单 | 已支付，等待接单者接单 |
| 已接单 | 接单者已接单，准备去取货 |
| 配送中 | 已取到货，正在配送 |
| 待确认 | 已送达，等待发单者确认 |
| 已完成 | 发单者确认，款项结算 |
| 已取消 | 订单取消 |
| 申诉中 | 发生纠纷，平台介入 |

### 5.4 支付结算流程

```
发单者下单
  │
  ▼
调用微信支付，款项进入平台商户号
  │
  ▼
订单完成
  │
  ▼
平台扣除服务费
  │
  ▼
剩余金额进入骑手账户余额
  │
  ▼
骑手申请提现
  │
  ▼
管理员审核后，转账到骑手微信零钱
```

## 9. 风险与难点

### 9.1 已知风险

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 微信支付需企业资质 | 个人无法收款 | 先用模拟支付/余额流程验证；正式运营需注册商户 |
| 接单者取消/跑路 | 发单者体验差 | 引入骑手认证+押金+信用分机制 |
| 发单者恶意不确认 | 骑手无法收款 | 送达后 24 小时自动确认 |
| 订单纠纷 | 平台运营压力大 | 后台申诉入口 + 客服介入 |
| 个人信息合规 | 学生证/身份证上传 | 明确隐私协议，仅用于认证 |

### 9.2 技术难点

1. **微信支付对接**：uniCloud 下对接微信支付需要正确配置商户号、证书、回调。
2. **并发抢单**：多个骑手同时抢单时，需要用数据库事务或原子操作防止超卖。
3. **自动结算**：订单完成后自动计算平台费和骑手收入，需保证幂等性。
4. **消息通知**：订单状态变更时及时通知双方，需配置微信订阅消息。
5. **后台权限控制**：admin 后台需防止普通用户越权访问。
